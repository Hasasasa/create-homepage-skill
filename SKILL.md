# Create Homepage Skill

把**一个具体的人物**做成一个**有辨识度、可运行、可选择上线**的个人主页。

这套流程不是凭空提炼，是从一次真实的 9 天创作（先做能跑的基础版 → 一轮轮按反馈和参考迭代 → 上线）里抽出来的。它的两个支点：

- **一套与隐喻无关的「连接层」**（设计 token + 可移植动效原语 + 按实际使用层覆盖的减动效/性能契约）——保证任何人一上来就有**协调、不空洞**的底子；是否动态由方向与技术路径决定。
- **一层专属外壳**（隐喻**或**直白编辑型方向、版式结构、视觉语言、文案）——每个人从**自己的证据、经确认的设定和参考**里长出来，绝不预置成模板。

底子由 skill 保证质量，外壳由这个人保证独特。这就是它既协调又不容易同质化的原因。

## 铁律（先读，贯穿全程）

1. **接住 brief，别当白纸访谈。** 用户通常带着身份、大致方向、甚至技术栈和材料就来了——先**接住并复述**你听到的（"你是 X，想要 Y 气质，5 个板块……对吗？"），别假装他是白纸、从零开始盘问。同时**探测你所在环境里可复用的 skill**（设计/动效/文案；环境无 skill 机制就跳过），少造轮子。见 `references/interview.md`。
2. **先处理已知参考和最小内容集，再出能跑的基础版（base-first）。** 用户已给参考时，先拆清“喜欢什么 / 不抄什么 / 如何迁移”，再做首版，避免明知方向却先造通用版；没有参考时才用中性、低成本的基础版建立反馈。改现有站默认保留原栈；新站选择最小合适技术路径。首版前只需身份、一句话定位、2–3 条代表性证据或经确认的设定、语言/隐私/技术约束，不要求补齐全部素材。
3. **隐喻先找后造，且隐喻可选。** 先用用户参考；没有参考再引导寻找；实在没有才共创。职业名称本身不算证据，必须有个人经历、作品材料、经确认的人物设定、受众任务等至少两个独立信号，且隐喻能帮助理解。始终保留一个不靠隐喻的编辑型候选。编辑型方向靠最强证据或设定、排版和一个**签名表达**建立识别度；签名表达可以是静态构图、字体处理、内容结构，也可以在任务需要时是交互。
4. **动效从库现取 + 套连接层，但不为动效反选重栈。** 先按任务选技术路径，再决定是否需要动效；静态/轻交互路径用 CSS 或少量原生 JS，React 强交互路径才按需从 `reactbits.dev` / `shaders.paper.design` 取。看懂原理后移植进已选栈，**不预置固定动效引擎**。每个取来的动效都必须过连接层：统一 token，并补上减动效降级与性能保护。见 `references/motion-kit.md`、`references/inspiration.md`。
5. **证据大于修饰，但只接收有权披露的最小材料。** 真实人物优先使用已脱敏摘录、公开资料或用户自己的总结；不要默认索要原始合同、聊天记录、会议纪要等受保密约束材料。确需处理源文档时，先确认用户有权分享，只提取必要事实，避免写入日志/搜索请求/公开仓库，并清理本轮临时副本。虚构人物以用户确认的设定为事实来源，不把模型推测擅自写成既定经历。正文定稿前再做去 AI 腔检查。
6. **减动效必须全静态降级（覆盖实际使用到的层，无障碍底线）。** 用了 CSS 动画/过渡时，以全局 `@media (prefers-reduced-motion: reduce)` 兜底；用了 JS 动效，再由该栈的 reduced-motion API 返回静态元素；用了 Canvas/WebGL，再用 `matchMedia` 停止渲染或换静帧。完全无动效的站不必添加无作用的技术层，也不要为了凑“齐三层”引入项目原本没有的 JS、Canvas 或 React。见 `references/motion-kit.md`。
7. **隐私按公开级别管理，不一刀切。** 开工前把信息分成：可公开原名、需泛化、禁止出现。公开雇主、公开项目和用户明确批准的联系方式可以保留；客户秘密、未公开项目、私人电话等按清单泛化或删除。公开交付前还要检查图片/分享卡元数据和构建产物，避免 EXIF/GPS、`.env`、令牌或服务端密钥泄漏。只有用户明确授权后才公开站点或源码。
8. **检查点分层验证，不是每次都跑上线项。** 基础版检查真实渲染、溢出、控制台和内容可见；上线元数据、链接和分享卡只在部署前检查。优先复用现有浏览器测试工具或环境能力，临时验证资产放在隔离位置，只清理本轮新增内容。见 `references/build-verify-deploy.md`。
9. **主动避开 AI 通病版式。** 没有明确要求时，别把独特性花在这些默认款上：暖米色 + 衬线 + 陶土色、纯黑配单一荧光色、Inter/Space Grotesk 当"安全牌"、处处 emoji 分节、全部居中、rounded-lg 满屏、紫蓝渐变 hero。用户明确要某一种就照做，否则从个人证据、经确认的设定和视觉方向里长出专属表达。**另守两条排版硬底线**（详见 `references/motion-kit.md` 第 2a 节）：① 中文大标题禁用为拉丁字母调的负字距 / `<1` 行高 / uppercase，会叠字；② 任何非系统通用字体必须有获授权、跨平台的 Web 加载路径，许可允许时可自托管 woff2，否则使用获授权的字体服务；别裸依赖 `Arial Narrow / Bahnschrift / Cascadia` 等本机字体。浏览器验证要核实际来源，部署前要核字体许可。
10. **过三层防同质化自检，不只换名换色。** 每定一版方向、每做完一版都检查：① **换人测试**：只换名字和主色，是否还能套给别人；② **去皮测试**：拿掉配色、字体、职业文案后，信息顺序、Hero 构图、导航与签名表达是否仍和常见方案/其它候选同骨架；③ **职业刻板测试**：概念是否只是职业名词的第一联想，而没有个人经历、作品证据、受众任务等至少两个彼此独立的信号。任一失败就重做。多个方向至少要在**信息主次与版块顺序 / Hero 构图 / 访问者任务与签名表达 / 贯穿动线与视觉语法**中的两项真正不同；换配色、换字体、换职业道具不算新方向。详见 `references/inspiration.md`。
11. **先决定要不要动；启用动效后再分主次、给预算。** 静态方向允许全站零动画。选择动态方向时，每个动效先派一个角色（定向 / 反馈 / 解释 / 签名 / 氛围），再守总量：普通区块最多轻入场或必要状态反馈，**全站只留 2–3 个高光签名动效、一屏最多一个高注意力效果、全局最多一层持续氛围底**。主次分明才显高级，满屏乱动就是噪音。预算细则见 `references/motion-kit.md` 第 7 节。

## 流程

> **进度清单**（建议复制进你的回复，逐项勾选）：
> `[ ] 0 接住 brief + 探测本地 skill` `[ ] 1 出基础版→冒烟检查→停下给用户看/选方向` `[ ] 2 收敛专属方向(参考/编辑/隐喻)` `[ ] 3 按反馈迭代 + 证据 + 一次 grilling` `[ ] 4 动效来源 + 连接层落地` `[ ] 5 隐私复核 + 全量 QA + 可选上线`

### 阶段 0 · 接住 brief + 探测本地 skill
1. **接住并复述用户的 brief**：身份、气质、参考、板块、现有技术栈/是否新建、语言，以及交付范围（仅本地 / 私有仓库 / 公开站）。已有参考就在首版前完成三点拆解；缺的关键项一次问一个。清单见 `references/interview.md`。
2. **探测可复用的 skill**（仅当你所在环境有 skill 机制时）：优先使用当前环境已经提供的技能目录、清单或检索能力，挑出**美化/设计/动效/去 AI 腔文案**相关项；只有环境没有目录能力、但明确允许读取本地 skill 路径时才查看对应目录。不要假设所有 agent 都使用同一安装位置。方法见 `references/inspiration.md` 的“复用本地 skill”。
3. 按 `references/build-verify-deploy.md` 定**技术路径**：改站保留原栈；新站选静态/轻交互或强交互路径。锁定隐私级别、技术路径、语言和最小内容集即可进入阶段 1；没有参考时，视觉/动效/隐喻不必定死。

### 阶段 1 · 先出能跑的基础版（base-first）
用选定技术路径搭基础版。用户带了参考，就按已拆解的方向实现；没有参考，就保持结构和视觉成本可调整。先铺共享 token；实际启用动效时，再补对应层的减动效和性能契约。静态路径用 CSS 变量与必要的原生能力，强交互路径才使用对应 motion/Canvas 实现。用最小内容集做出协调、能跑的首版，其他材料可明确占位。

**这里是一个强制停顿点，别一口气把 base → 打磨 → 全量 QA 连成一次性交付。** 首版只跑**单视口冒烟检查**（无横向溢出 + 0 报错 + 首屏肉眼过一遍，见 `references/build-verify-deploy.md`），就停下给用户看，同时给 **2–3 个具体方向候选**让他选或否决、问清一两条"必须要/绝对不要"的点。风格输入靠"对着具体雏形反应"拿到，不做抽象风格问卷（那是铁律 1 要避免的白纸访谈）。拿到方向后再投入打磨与最终版的全量多视口 QA——方向没定就跑全量验证，一旦推倒方向那轮 QA 全白做，也是"跑很久才出结果"的主因。

### 阶段 2 · 收敛专属方向（参考 / 编辑 / 隐喻）
基础版给用户后继续收敛**专属视觉方向**，隐喻只是候选之一；若用户首轮已给参考，这里基于成品反馈迭代，不再从头找方向：
1. **先找**：若阶段 0 没有参考，再问有没有想模仿的站/截图/组件。有则按 `references/inspiration.md` 三点拆解后再改版。
2. **引导找**：没有 → 让他去 Awwwards / reactbits / paper 逛一圈，带回 2–3 个"想要这种感觉"的例子。
3. **才共创（且先判断需不需要隐喻）**：实在没有参考 → 先看这个人**撑不撑得起隐喻**。职业身份本身不算证据；必须有个人经历、作品材料、经确认的人物设定、反复出现的动作、受众任务等至少两个彼此独立的信号，而且隐喻确实帮助理解。撑得起 → 推导 2–3 个候选，并同时给一个不靠隐喻的编辑型候选，避免被职业第一联想锚定；撑不起 → **走直白的编辑型方向**：用最强证据或设定定内容主次 + 有性格的排版构图 + 一个签名表达（可静态），不套主题外壳。两条路都过铁律 10 的三层自检——无隐喻 ≠ 平庸。

无论走哪条路，产出的方向候选都必须**过铁律 10 的三层自检**。候选至少在信息结构、Hero 构图、访问者任务/签名表达、贯穿动线/视觉语法中的两项不同；不要拿同一套“大标题 + 右侧演示器 + 微型英文标签 + 网格线”只换职业道具。定稿也要做一次同职业反事实：想象另一个同职业、但经历与受众不同的人，若仍能原样套用，就是职业模板。

### 阶段 3 · 按反馈迭代 + 证据 + 一次 grilling
这是真实质量引擎，允许推倒重来：
- **接住直白的审美否决就改**（"太商务""更丑了""太单调"这类反馈比正面 brief 更能逼出好版本）；自己看页面给具体诊断，别把设计题甩回去问。
- **补深度证据或设定**（见铁律 5、`references/asset-checklist.md`）：真实人物优先公开来源和已脱敏摘录，虚构人物沿用用户确认的设定；把通用套话换成有来源的具体内容。只有数据可靠且值得维护时才接 live 数据。
- **当已有具体成品、但仍"太单调"时，跑一次结构化 grilling**（`references/interview.md` 的问题树）：把零散抱怨收敛成**一个连贯的大改**（真实项目里就是这样定下"继续飞行"的滚动叙事），而不是打无数零碎补丁。

### 阶段 4 · 动效来源 + 连接层落地
需要某个具体动效时，先问它是否服务访问者任务，再按已选技术路径实现：静态/轻交互站优先 CSS 或少量原生 JS；React 强交互站可从 `reactbits.dev` / `shaders.paper.design` 按需取（见 `references/inspiration.md`），看懂原理后移植。每个动效都要过连接层：统一 token，补 `prefers-reduced-motion` 静态降级和相应性能保护，并守动效预算。React/Canvas 的具体规格见 `references/motion-kit.md`。

### 阶段 5 · 隐私复核 + 可选上线
先按交付范围复核隐私与基础检查。用户只要本地成品时，到本地可运行并完成基础验收即可，不强制补齐公开站元数据。用户要求部署准备或实际部署时，再补 SEO meta、分享卡、favicon、必要的结构化数据并运行部署前检查；只有明确授权对应外部动作后，才沿用现有 VCS/仓库/托管，或创建新仓库、选择公开/私有、部署和绑定域名。公开源码与公开网站分别确认。见 `references/build-verify-deploy.md`。
