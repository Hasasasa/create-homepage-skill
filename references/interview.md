# 接住 brief + grilling 问题树

分两部分：**阶段 0 先接住用户带来的 brief**（别当白纸从零盘问）；**阶段 3 在已有基础版之后，才用 grilling 问题树**做一次连贯的大跃迁。

> **核心原则：所有"推荐答案"必须从当前用户材料里推导，不是标准答案。** 每个人的世界观、审美、想要的动效都不同——目标是帮他做出**适合他自己**的站，不是把某一版成品复制给下一个人。凡涉及视觉/动效/交互，一律先根据用户的身份、内容、受众、参考提出 2–3 个**专属方向**，或让用户拿出自己的参考，再据此实现。

---

## 阶段 0 · 接住 brief（复述清单）

用户来的时候通常已经带了不少信息。你的第一步是**把听到的复述一遍让他确认**，而不是假装他什么都没说。对照下面这份清单，**他已经给的就复述确认，缺的关键项一次问一个**：

- **身份**：职位/定位、一句话是谁、经验年限（问清口径：总工龄 ≠ 某岗位年限）。
- **想要的气质/风格方向**：暗色/亮色？高级克制 / 张扬 / 极简 / 科技感？（真实项目里用户一句"暗色系、高级、克制、有科技感，但不要像模板"就定了大方向。）
- **有没有参考**（关键，决定阶段 2 的路径）：想模仿的站/截图/现成组件代码？有就记下来，阶段 2 以它为准。
- **板块结构**：他心里有没有大致的分区（Hero / 关于 / 项目 / 能力 / 联系）？有就先按他的来。
- **技术栈 / 约束**：指定框架？版心宽度？只做 PC 还是要移动端？
- **语言语气**：中文 / 双语；正式书面 / 口语化。
- **硬约束**：**脱敏**（哪些真实客户/雇主/项目/电话不能出现）、部署目标。

**只锁硬约束（脱敏、技术栈、语言）就够进阶段 1。** 气质、板块、动效、隐喻这些，先出基础版再在迭代里收敛，不必现在盘问到底。

---

## 阶段 3 · grilling 问题树（已有基础版之后再用）

**什么时候用**：基础版做出来了，用户看了觉得"能用但太单调 / 不够是我的风格"，但又说不清要改哪。这时用下面的问题树，把零散的不满**收敛成一个连贯的大改**（真实项目里，正是一次这样的 grilling 定下了贯穿全站的"继续飞行"滚动叙事，而不是打无数补丁）。

**怎么用**：一次只问一个问题、给出从用户材料推导的推荐答案、等确认再问下一个。一次抛一堆会让人懵。能从已有资料/代码查到的*事实*自己查，只把*决策*交给用户。

### Q1 · 世界观 / 隐喻（先定，统领全站；且隐喻可选）
先走"先找后造"（见 `inspiration.md`）：有参考按参考来。没有则**先判断这个人撑不撑得起隐喻**——身份/内容/受众 ≥2 个信号支持、且隐喻能帮人看懂，才从职业/故事/关键词提炼 2–3 个候选，说清每个怎么渗透进文案/章节名/图标/动效；撑不起就走**直白的编辑型方向**（证据结构 + 排版性格 + 一个签名交互），不硬套主题外壳。**不预设某职业只能对应某隐喻，也不强求每个人都要有隐喻**——两条路都过铁律 10 的换名换色测试，无隐喻 ≠ 平庸。

### Q2 · 主结构
"单页滚动叙事，还是分区块/多屏？" 推荐滚动叙事单页——最适合承载连续动效与故事感；用户要传统分区也可以。

### Q3 · 贯穿动线
"用什么把各板块串成一条线，而不是几个孤立区块堆叠？" 从已锁定的方向推导一条主动线（一个随滚动推进的意象/进度/路径）：有隐喻就从隐喻长出来（真实项目里的"继续飞行"）；走编辑型方向就从内容主线/证据强度长出来（如按时间或论点推进的一条脊）。这是把"单调"变"连贯"的关键一问，也是治结构同质的主要抓手。具体效果去库里找（见 `inspiration.md`），落地过连接层。

### Q4 · Hero 主体
"首屏正中放什么当主角？名字/标题怎么处理？" 给 2–3 个从世界观、姓名/handle、主身份推导的方案，说清它如何承接下面的动线。别默认某种固定形态。

### Q5 · 背景层
"要不要一层贯穿全站的氛围背景？动态 / 静态 / 图像 / 纹理 / 网格 / 无？" 需要"活"的感觉时用一层低成本动态底（从库取粒子/网格视觉，套连接层的 `AmbientCanvas` 契约包装：触屏减半、切后台暂停、减动效不渲染）；追求极简或内容密度高时用静底。别为炫技默认上粒子。

### Q6 · 微交互清单
"这几种微交互想要哪些？逐个挑，别默认全要。" 先从内容任务推导：访问者要复制联系方式、切换作品类别、理解时间线、进入外部频道、感知实时成果？**只提能服务这些任务的交互**，挑 2–4 个，每个都要能回答"它在这个站里解决什么"。

### Q7 · 动效手感
"整体手感偏哪种？弹簧回弹 vs 克制线性、快 vs 慢、张扬 vs 内敛？" 动效库锁定 framer-motion（连接层）。手感由用户定；带了参考站就照它调 `theme.js` 的 spring 预设。

### Q8 · 时间线 / 项目形态
"经历/项目怎么排？" 先看内容结构再定形态：有明显主线/副线就做可切换叙事；作品比时间重要就按主题/证据强度组织；要极简就用列表。**别拿固定形态倒套内容。**

### Q9 · 内容取舍与证据（proof over polish，别跳过）
"这些经历/项目里，哪 **2–3 个**最能证明你的价值？" 个人站最露馅的是通用套话、不是布局：
- 每条逼出**可量化成果**（数字、上线/获奖/增长），不是职责罗列。
- **角色边界如实**：具体做了什么，别把团队成果写成个人的。
- 宁可精选几条强证据，也不堆一长串平淡条目。
- 空泛形容词（"资深""核心""赋能"）换成具体事实。所有正文定稿前过去 AI 腔检查（装了 `avoid-ai-writing` 就用它，否则照其原则自查）。
- 主动向用户要**真实源文档**抠这些证据（见 `asset-checklist.md`）。

### Q10 · 实时数据
"要不要接实时数据徽标（GitHub star、订阅数、App 评分）？" 有数据源就接（双源 + 本地缓存 + 兜底值），比写死数字可信、会自更新；没有就跳过。

### Q11 · 移动端策略
"移动端叙事全保留还是简化？" 推荐叙事全保留、背景降密、卡片改竖排；**减少动效偏好下全静态**（无障碍底线）。

### Q12 · 文案语气 & Q13 · 脱敏
语气正式书面 vs 口语化、中文/双语，必须确认。脱敏在阶段 0 已锁，这里复核：真实客户/雇主/项目名/电话一律泛称，公开部署全程遵守。

---

grilling 走完后，向用户**复述这一次大改的完整方案**（世界观 + 动线 + 各项决策 + 用到的参考），确认"就按这个改"，再动手。
