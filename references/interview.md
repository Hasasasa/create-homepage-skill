# 接住 brief + grilling 问题树

分两部分：**阶段 0 先接住用户带来的 brief**（别当白纸从零盘问）；**阶段 3 在已有基础版之后，才用 grilling 问题树**做一次连贯的大跃迁。

> **核心原则：所有推荐都从当前材料推导。** 用户已给参考时，先拆参考再实现首版；没有参考时，不要求在开工前凭空定完 2–3 个方向，可先用低成本、中性基础版建立反馈。需要收敛方向时，再基于身份、证据、受众和反馈提出候选。

---

## 阶段 0 · 接住 brief（复述清单）

用户来的时候通常已经带了不少信息。你的第一步是**把听到的复述一遍让他确认**，而不是假装他什么都没说。对照下面这份清单，**他已经给的就复述确认，缺的关键项一次问一个**：

- **身份**：职位/定位、一句话是谁、经验年限（问清口径：总工龄 ≠ 某岗位年限）。
- **想要的气质/风格方向**：暗色/亮色？高级克制 / 张扬 / 极简 / 科技感？（真实项目里用户一句"暗色系、高级、克制、有科技感，但不要像模板"就定了大方向。）
- **有没有参考**（关键，决定首版路径）：想模仿的站/截图/现成组件代码？有就立即按“喜欢什么 / 不抄什么 / 如何迁移”拆解，再进入基础版；阶段 2 只基于成品反馈继续收敛。
- **板块结构**：他心里有没有大致的分区（Hero / 关于 / 项目 / 能力 / 联系）？有就先按他的来。
- **技术路径 / 约束**：新建还是改现有站？已有框架和样式体系是什么？用户是否指定技术栈？交互复杂度是否真的需要运行时框架？版心宽度？只做 PC 还是要移动端？
- **语言语气**：中文 / 双语；正式书面 / 口语化。
- **隐私与交付范围**：哪些信息可公开原名、需泛化、禁止出现；只做本地、进入私有仓库，还是准备公开站点。公开源码、部署、绑域名分别授权。

**锁定隐私级别、技术路径、语言和最小内容集就够进阶段 1。** 最小内容集是身份、一句话定位和 2–3 条代表性证据或经确认的设定；其余素材不要一次问完。改站保留原栈；新站选择最小合适路径。没有参考时，气质、动效和隐喻可在首版后收敛。

## 阶段 3 · grilling 问题树（已有基础版之后再用）

**什么时候用**：基础版做出来了，用户看了觉得"能用但太单调 / 不够是我的风格"，但又说不清要改哪。这时用下面的问题树，把零散的不满**收敛成一个连贯的大改**（真实项目里，正是一次这样的 grilling 定下了贯穿全站的"继续飞行"滚动叙事，而不是打无数补丁）。

**怎么用**：一次只问一个问题、给出从用户材料推导的推荐答案、等确认再问下一个。一次抛一堆会让人懵。能从已有资料/代码查到的*事实*自己查，只把*决策*交给用户。

### Q1 · 先判断是否需要世界观 / 隐喻
先走"先找后造"（见 `inspiration.md`）。职业名称本身不算证据；个人经历、作品材料、反复动作、受众任务等至少两个独立信号支持且隐喻能帮助理解，才提炼候选，并同时给一个编辑型候选。编辑型方向使用**签名表达**建立识别度，它可以是静态版式、文字结构或交互。两条路都过换人 / 去皮 / 同职业反事实自检。

### Q2 · 主结构
"单页连续叙事，还是分区块/多页？" 根据访问者任务、内容关系和浏览深度给推荐：需要连续论证时可用滚动叙事，需要快速扫描或内容彼此独立时优先清晰分区/多页。不要为了故事感默认选择滚动叙事。

### Q3 · 贯穿动线
"用什么把各板块串成一条线，而不是几个孤立区块堆叠？" 从已锁定的方向推导一条主动线：它可以是静态的信息递进、时间/论点结构，也可以在动态方向中成为随滚动推进的意象、进度或路径。有隐喻就从隐喻长出来（真实项目里的"继续飞行"）；走编辑型方向就从内容主线/证据强度长出来。只有确实需要动态效果时才去库里评估，并按 `inspiration.md` 过连接层。

### Q4 · Hero 主体
"首屏把什么当主角？名字/标题怎么处理？" 给 2–3 个从个人证据、经确认的设定、编辑方向、姓名/handle 或主身份推导的方案，说清它如何承接下面的信息结构。别默认正中、左右分栏或某种固定形态。

### Q5 · 背景层
"要不要一层贯穿全站的氛围背景？动态 / 静态 / 图像 / 纹理 / 网格 / 无？" 先判断背景是否帮助内容层级；多数情况下静底、图像或纹理已经够用。确实需要动态氛围时才从库评估，并套连接层的 `AmbientCanvas` 契约包装：触屏减负、切后台暂停、减动效不渲染。别把 Canvas/WebGL 说成天然低成本，也别为炫技默认上粒子。

### Q6 · 微交互清单
"这几种微交互想要哪些？逐个挑，别默认全要。" 先从内容任务推导：访问者要复制联系方式、切换作品类别、理解时间线、进入外部频道、感知实时成果？**只提能服务这些任务的交互**，挑 0–4 个，每个都要能回答"它在这个站里解决什么"；没有对应任务时，0 个是有效答案。

### Q7 · 动效手感
先问“这个方向需要动效吗？”不需要就全静。需要时再问“整体手感偏弹簧回弹还是克制线性、快还是慢、张扬还是内敛？”按已选技术路径实现：静态/轻交互站用 CSS 或少量原生 JS；React 强交互站沿用现有或选择兼容的 Motion 方案，并在项目共享 token 源中调整预设，不写死包名或文件名。

### Q8 · 时间线 / 项目形态
"经历/项目怎么排？" 先看内容结构再定形态：有明显主线/副线就做可切换叙事；作品比时间重要就按主题/证据强度组织；要极简就用列表。**别拿固定形态倒套内容。**

### Q9 · 内容取舍与证据（proof over polish，别跳过）
"这些经历/项目里，哪 **2–3 个**最能证明你的价值？" 个人站最露馅的是通用套话、不是布局：
- 每条逼出**可量化成果**（数字、上线/获奖/增长），不是职责罗列。
- **角色边界如实**：具体做了什么，别把团队成果写成个人的。
- 宁可精选几条强证据，也不堆一长串平淡条目。
- 空泛形容词（"资深""核心""赋能"）换成具体事实。所有正文定稿前过去 AI 腔检查（装了 `avoid-ai-writing` 就用它，否则照其原则自查）。
- 优先从**公开页面、已脱敏摘录或用户自己的事实总结**提取证据；确需源文档时先确认披露权限，并只处理必要片段（见 `asset-checklist.md`）。

### Q10 · 实时数据
"要不要接实时数据徽标（GitHub star、订阅数、App 评分）？" 只有数据公开、来源可靠、维护成本值得且访问者确实需要时才接。明确加载、失败和过期状态；缓存值要带更新时间，不能用无来源的“兜底数字”伪装实时结果。没有可靠数据源就跳过。

### Q11 · 移动端策略
"移动端叙事全保留还是简化？" 按移动访问者的主要任务决定信息保留与顺序，不能只把桌面卡片机械改竖排。核心证据和主要操作保留，装饰背景与高成本动效可降级；**减少动效偏好下全静态**（无障碍底线）。

### Q12 · 文案语气 & Q13 · 脱敏
确认正式/口语、中文/双语。隐私在阶段 0 已分级，这里复核：可公开项保留事实，需泛化项使用批准的替代称呼，禁止项删除；不得自行扩大公开范围。

---

grilling 走完后，向用户**复述这一次大改的完整方案**（视觉方向 + 信息/叙事结构 + 各项决策 + 用到的参考；有隐喻时再说明世界观），确认“就按这个改”，再动手。
