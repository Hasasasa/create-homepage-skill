# 素材与账号清单（向用户索取）

硬约束锁定后，把下面清单发给用户让他补齐。缺什么当场要，别边做边猜。所有文案/数据最终集中到 `src/data/site.js` 一个文件，视觉决策集中到 `src/data/theme.js`（见 `motion-kit.md`）。

## 内容素材

- **身份**：中文名、英文名 / handle、一句话定位（职位）、经验年限（如需拆分：X 年项目管理 + Y 年产品，问清口径——"总工龄"和"某岗位年限"是两回事）。
- **关于我**：3 段左右的介绍，或给要点让你起草（起草后必须回给用户确认语气）。
- **经历 / 项目时间线**：每条给 `时间 / 标题 / 角色 / 一句话描述 / 链接`。正职与副业/开源**分开给**（对应主线/支线）。**优先给能证明价值的少数几条**，每条带**可量化成果**（数字、上线、增长），角色如实不夸大——proof over polish，见 `interview.md` Q9。
- **实时数据源**：GitHub 用户名（拉全仓库 star）、要展示哪几个仓库；其它想接的实时徽标。
- **社交与联系**：平台 + 链接（B站/YouTube/GitHub/App Store/小红书……）、联系邮箱。
- **图片 / logo**（可选）：文件路径。**提醒**：会随代码进公开仓库，涉及肖像/隐私要用户确认。用不到就别塞进 public（每次构建都打包）。
  - **先分诊再锁布局**：拿到用户的图，逐个按角色归类（hero 人像 / 作品截图 / logo / 二维码 / 纹理底 / 产品视频），并标**能不能用 + 原因**——例如"小尺寸截图铺满宽 hero 会糊，不可用""竖图当横 banner 不合比例"。不可用的：让用户换图、改布局迁就、或用生成式封面/纯色兜底。别拿到图就硬塞进设计里。

## 真实源文档（主动要，这是"证据大于修饰"的来源）

个人站最露馅的是通用套话。真实项目里，内容之所以可信，正是因为从用户的**真实文档**里抠出了可核实的数字和事实。主动问用户有没有下面这些，能给就给：

- **项目结题 / 复盘 PPT、需求文档、验收报告** → 抠**可量化成果**（"出图效果达竞品 80%""150+ 设计师使用""合并 11 个 PR / 7910 行代码"）。
- **会议纪要 / 智能纪要 / 聊天记录** → 还原他在项目里**具体做了什么、边界在哪**。
- **合同 / 立项材料** → 核实角色、周期、规模。
- **live 数据源**（GitHub API、App Store lookup、ungh.cc）→ 接实时徽标，比写死数字更可信。

拿到后：把职责罗列换成结果、把空泛形容词换成事实、把团队成果和个人贡献分清。**注意**：这些文档往往含敏感信息，抠完证据后按脱敏清单泛化，原文不入库。

## 规则类（必须写下来）

- **脱敏清单**：哪些真实客户名/雇主名/项目名/电话不能出现，各自的泛称替代（"某新能源车企""某头部车企""AI 科技公司"）。公开部署全程遵守。
- **文案语气**：正式书面 / 口语化；语言（中文 / 双语）。定稿前所有正文过一遍去 AI 腔检查（装了 `avoid-ai-writing` 就用它，否则照其原则自查）。
- **视觉偏好**：主色/强调色、从参考站带回的 2–3 个例子；想模仿的动效参考（站链接 / 效果截图 / 现成组件代码），逐个标"想要它的哪一点"。

## 部署相关（阶段 5 才用，可先问清）

- **域名**：有没有自己的域名？想不想买？DNS 托管在哪？（**若目标访客在中国大陆**，建议 Cloudflare Pages + 自有域名，可达性较好；否则任意静态托管皆可。）
- **账号授权**（这两步必须用户本人在浏览器完成）：
  - GitHub：`gh auth login`（或确认已登录）。
  - Cloudflare：`npx wrangler login` —— 用户点浏览器里的 Allow 授权。
  - 绑定域名的 DNS 记录 / 激活，需用户在 Cloudflare 面板操作。

## 事实类（自己查，别问用户）

- 已上架 App 的上线时间、评分 → App Store lookup API（`itunes.apple.com/lookup?id=...`）。
- 仓库的 star / 创建时间 / 描述 → GitHub API 或 `ungh.cc`。
- 现有代码里的结构、样式、历史决策 → 直接读文件。
