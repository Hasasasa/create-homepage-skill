# 参考与动效来源

两件事：**隐喻/审美从哪找**（先找后造，且隐喻可选），**具体动效从哪取**（仅在所选技术路径需要时现取，并过连接层）。

无论哪条路，参考站**只用来校准气质与手感、或作为可移植的效果来源，不是拿来抄版式**——最终视觉必须从个人证据、经确认的设定或编辑方向里长出来，不强制先造世界观。

---

## 先探测复用本地 skill（先看，少造轮子）

动手前，**若你所在环境有 skill 机制，先看有没有能直接用的 skill**——很多美化/设计的活别人已经封装好了。环境没有 skill 机制就跳过本节。

**怎么探测：** 优先使用当前环境提供的技能目录、清单或检索工具，并遵循该环境的技能加载规则。只有没有目录能力、但明确允许读取本地 skill 路径时，才检查对应用户级/项目级目录并读取描述；不要假设固定使用 `~/.claude/skills/`、`~/.codex/skills/` 或其它单一路径。重点挑这几类：
- **设计系统 / UI 风格**（premium、expressive、professional、clean、bento、shadcn 等）
- **动效 / 组件**
- **去 AI 腔文案**（如 avoid-ai-writing）
- **Figma / 视觉生成**

**怎么复用：**
- 设计系统 skill → 给阶段 0 校准提供词汇和起点，把配色 / 字阶 / 间距 / 圆角约定写入所选技术路径的 token 来源；React 新站可以是 `theme.js`，其他项目沿用原约定。
- 文案 skill（如 avoid-ai-writing）→ 装了就在正文定稿前必过；没有就按去 AI 腔原则自查（铁律 5）。
- 动效 / 组件 skill → 作为实现技法参考，落地仍要过连接层；不要把现成模块外壳当作个人站结构。
- 探测到就告诉用户“环境里有 X，我会用它处理 Y”；只有该 skill 会改变交付范围、外部状态或关键设计选择时才额外确认，不为正常复用制造停顿。

## 隐喻 / 审美：先找后造

### 先找 —— 优先用户自带参考
真实项目里那个"航行/纸飞机"世界观，就是用户自己贴了 `paperplanes.world` 才出现的，不是访谈现场发明的。所以第一步是**确认用户是否已经提供想模仿的站/截图/组件代码**；有就先拆解并以它校准首版，没有再询问或引导寻找。

### 引导找 —— 没有就去这些站带回 2–3 个例子
- **awwwards.com** — 获奖网站画廊，可观察高表现力网页的大胆程度、滚动叙事结构、排版主次与留白节奏；它是灵感来源，不是通用可用性或转化率标准。
- **siteinspire.com** — 看**版式结构**。
- **pinterest / behance** — 看**气质与品牌调性**。
- 让用户挑 2–3 个向往的，逐个问"你喜欢它哪一点"——配色？字体？某个滚动效果？还是整体气质？

### 才共创 —— 实在没有才推导，且先判断需不需要隐喻
**隐喻是可选的，不是每个人都要有。** 先看这个人**撑不撑得起隐喻**。职业名称本身不计为支持信号；至少要有两个彼此独立、来自职业之外的信号，例如：个人经历或经确认人设中的转折、作品反复出现的材料/动作、证据材料的结构、访问者要完成的任务、人物惯用的语言。两个同义关键词不算两个信号。

- **撑得起** → 从上述个人证据或经确认的设定提炼 2–3 个候选，同时给一个不靠隐喻的编辑型候选。模型最先想到的职业联想默认是高风险项；除非具体信号能把它变得个人化，否则舍弃。
- **撑不起** → **走直白的编辑型方向**：不套主题外壳，独特性来自最强证据、字体排版、留白构图和一个签名表达；签名表达可以完全静态。

**该拒绝隐喻的信号**：纯为了显得聪明、把职业刻板化、让项目证据更难扫读、需要用户根本不会说的装饰性文案。两条路最后都要过铁律 10 的**换人 / 去皮 / 同职业反事实**三层自检——无隐喻 ≠ 平庸。

### 拆解每个参考（自带的或选中的）
拿到一个参考，**别整体照搬**——逼自己拆成三点写下来：
1. **喜欢它哪一点**：具体到配色 / 某个滚动效果 / hover 手感 / 排版气质，不是"整体好看"。
2. **绝不照抄哪一点**：明确一条不碰的，避免站变成它的翻版（也顺带避开它可能带的 AI 通病版式）。
3. **可迁移到这个人身上的想法**：把那“喜欢的一点”翻译成其证据、设定或编辑方向中的表达。

**收尾自检（铁律 10）**：
1. **换人**：换掉名字与主色还能套给别人吗？
2. **去皮**：去掉配色、字体与职业文案，是否仍是同一套 Hero、导航、信息顺序和交互骨架？
3. **同职业反事实**：换成另一个同职业、但经历/受众不同的人，是否仍可原样套用？

多个候选至少要在**信息主次与版块顺序 / Hero 构图 / 访问者任务与签名表达 / 贯穿动线与视觉语法**中的两项不同。签名表达可以是静态构图、排版或内容结构，也可以在任务需要时是交互。大标题左、概念演示器右、固定顶栏、微型英文标签、细网格线等元素并非禁用，但不能成套作为每个人的默认骨架。

## 具体动效：按技术路径现取

静态/轻交互路径先用 CSS 或少量原生 JS。强交互路径需要外部组件时，先检查当前源码的 imports、peer dependencies、框架版本和许可证，再决定移植；不要假设示例与项目版本天然兼容。

> 优先使用来源和许可证清晰的组件。记录来源 URL、版本或 commit，并在取用当时核对、遵守完整许可证；`THIRD_PARTY_NOTICES.md` 不是自动合规证明。不要依赖本文件记忆某个项目的“当前许可证”，许可证可能随版本变化。

### reactbits.dev —— 首选，最贴栈
动画化 React 组件集。不同版本和组件可能使用 `motion/react`、Canvas/WebGL 或额外依赖，先读源码和安装说明再移植。
- Text animations：SplitText（逐字入场）、Scramble、Glitch、Decrypted、Shiny、Gradient——标题动效现成方案。
- Backgrounds：各种粒子/网格背景。
- 卡片辉光、磁吸、3D 倾斜等微交互。
- **移植注意**：核对 imports、peer dependencies、React 版本和许可证；覆盖实际用到的 CSS、JS、Canvas/WebGL 减动效层。保留要求的版权/许可证文本，并记录来源版本。

### shaders.paper.design —— 生成式 WebGL 背景/封面
Paper 提供 mesh / warp / grain 等生成式 shader，可评估为动态封面或氛围背景。安装前按当时官方文档核对包名、导出 API、框架兼容性和许可证，不把这里的历史示例当安装指令。
- **移植注意**：WebGL 较重，**懒加载、控制弱机开销、切后台暂停**。作品无真实截图时的动态封面、或全局氛围底很好用。

### 手感对标
需要弹簧手感时，可参考组件源码的 stiffness / damping / mass，并写入项目自己的 motion token；不要复制某个版本的数值当永久标准。

**来源速查**（不同站看不同维度）：reactbits 看**可评估、可适配的效果与微交互手感**、paper 看**生成式背景**、Awwwards 看 **hero 冲击力与张扬度**、SiteInspire 看**版式结构**、Pinterest/Behance 看**气质调性**。任何外部实现都要重新核对版本、依赖、许可证与当前技术栈，不把示例当成可直接复制的承诺。
