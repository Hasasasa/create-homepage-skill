# 技术栈 · 验证 · 上线就绪 · 部署 · 改版诊断

## 技术路径路由（先选最小合适方案）

先看用户是否已有站，再看交互复杂度；不要为了套本 skill 迁移技术栈，也不要为了一个淡入效果引入整套 React 动效栈。

| 场景 | 默认路径 | 原则 |
|---|---|---|
| **改造现有个人站** | 保留现有框架、构建和样式体系 | 做局部、可回滚的改动；只有用户明确要求迁移时才换栈 |
| **新建静态 / 编辑型 / 极简站** | HTML + CSS + 必要的原生 JS，或用户指定的静态方案 | 内容和排版优先；能用 CSS 完成就不引运行时框架 |
| **新建强交互站** | React + 与其兼容的现代构建工具和按需动效库 | 适用于多状态交互、滚动叙事、复杂可视化或多个签名动效；选当前稳定且彼此兼容的版本，不写死旧主版本 |
| **用户明确指定技术栈** | 遵从指定 | 在该栈内实现连接层契约，不擅自改栈 |

无论走哪条路径，都保留连接层的**契约**：共享设计 token；若实际启用动效，再补动效角色/预算、`prefers-reduced-motion` 静态回退和性能保护。实现不必相同：静态路径以 CSS 变量为基础，只在任务需要时加入 CSS 过渡或少量原生 JS；React 强交互路径可使用 `motion-kit.md` 的现成组件。

### React 强交互路径

下面这组组合来自真实上线项目，只有选中 React 强交互路径时才按需使用：

- **React + Vite**——新项目采用当前稳定且相互兼容的版本；改造项目沿用现有主版本，除非用户明确授权升级。
- **React 动效库**——优先沿用项目已有方案；新引入时按当前官方包名和入口使用。若所选版本支持 `LazyMotion` 一类按需能力，则启用轻量加载。具体动效契约见 `motion-kit.md`。
- **GSAP + ScrollTrigger —— 按需选，不与上一条互斥**：需要**时间线编排的开场序列**（遮罩揭开 / 位移 / 压缩归位）或**滚动驱动的 pin / scrub / reveal / parallax**（高端作品集、创意机构官网那一类）时，用 GSAP + ScrollTrigger 通常比声明式动效库顺手；组件级入场、手势、布局动画仍用 Motion，两者可共存。契约照样落：`prefers-reduced-motion` 下不注册 ScrollTrigger（或 `.kill()`）、组件卸载时 `ctx.revert()` / kill 掉 trigger、字体与图片加载完成后 `ScrollTrigger.refresh()` 校正触发点。按需引入，别塞进首屏关键 bundle。
- **Canvas 2D**——环境背景层、点阵/粒子类效果（视觉从库现取，套 `motion-kit.md` 的 `AmbientCanvas` 性能契约包装）。
- **three.js —— 仅当需要 3D 时**才引，且**动态 import + 只在用到的那个组件里**（真实项目里只有开屏地球用它，主站不含），失败要有降级回退。绝不进主 bundle。
- **生成式 WebGL 组件 —— 按需**：确有背景/封面需求时，按当时官方文档核对包名、导出 API、框架兼容性和许可证后再安装；懒加载并控制弱机开销，不把历史包名写成永久指令。
- **纯 CSS + CSS 变量**（`:root` token），小站可用单一 `index.css`；避免只为个别组件引入整套 Tailwind。
- **数据驱动**：把文案/数据与视觉 token 各自集中到项目合适的单一来源；具体文件名和目录跟随现有架构，不把 React 项目结构当作跨技术栈契约。

设计 token 规格（颜色、字阶、间距、动效预设）见 `motion-kit.md` 第 2 节——**spec 先行、代码在后**，在所选技术栈的共享样式或 token 源中统一定义，避免 build 漂移成"处处略不一致"的通病。

引入依赖或外部组件前，先核对实际包名与 import 入口、框架支持范围、`peerDependencies`、构建/SSR 条件和许可证；沿用项目锁文件与包管理器，并保留许可证要求的声明。不要因为示例看起来相近就承诺"复制即用"，也不要顺带升级现有项目的主版本。

## 真实渲染验证（检查点跑，不是每改一处都跑）

把相关改动攒成一个完整变更，在有意义的节点（完成一个模块后、给用户看前、提交/部署前）跑一次。按以下顺序选择验证方式：

- 优先运行项目已有的 Playwright、Puppeteer、Cypress、端到端脚本或截图命令，不添加重复工具。
- 项目没有浏览器测试时，优先使用当前环境提供的浏览器控制能力连接本地预览。
- 只有前两者都不可用时，才在系统临时目录创建唯一命名的隔离验证目录并安装临时浏览器工具；不要改项目的 manifest、锁文件或依赖树。记录本轮创建的路径，结束后只清理这些临时脚本、依赖和缓存，绝不删除项目原有依赖或脚本。需要保留的截图/报告移到明确的 QA 产物目录。

无论使用哪种工具，都遵守以下渲染规则：

- **逐屏 `scrollIntoView` + 等待页面稳定**后再截图（hash 导航、懒加载和入场动画都需要真实触发；优先等待可观测状态，固定延时只作兜底）。
- 视口用正常尺寸（如 1440×900），**别用超高视口截全页**（`100vh` 会随视口膨胀）。
- 仅在当前浏览器工具支持且 WebGL 确实需要软渲染时，才加对应 SwiftShader/ANGLE 参数；同时保留一次接近真实用户环境的验证。

### 冒烟检查 vs 全量 QA（别在方向未定前跑全量）

验证分两档，按版本成熟度选，别每版都跑全量：

- **冒烟检查**（中间版本，尤其 base-first 首版）：**单视口**跑三项——无横向溢出、0 未处理报错、首屏肉眼过一遍。够判断"能不能给用户看、方向对不对"即可，几十秒的事。
- **全量 QA**（仅方向已确认的最终版跑一次）：下面「基础检查点」的完整清单——多视口（1440 + 390）逐屏 `scrollIntoView` 截图、逐分区肉眼过、CJK 叠字、字体来源、减动效、无障碍。**方向没定就跑全量，一旦推倒方向那轮截图全白做**，是"跑很久才出结果"的主要来源。

### 基础检查点（全量 QA 的完整清单）

1. 用实际滚动根节点检查**无横向溢出**，至少覆盖 1440 与 390 两个宽度：

   ```js
   const root = document.scrollingElement ?? document.documentElement
   root.scrollWidth <= root.clientWidth
   ```

2. **0 未处理页面错误，且没有由本次改动引入的相关控制台错误**（监听 `console`/`pageerror`；已知第三方噪音要记录来源，不能直接忽略）。
3. 逐屏截图肉眼过一遍：有没有错位、盖住、糊图、留白撑开。
4. **中文大标题不叠字**：放大看多行中文大标题有没有上下压字/碰撞（1440 与 390 都看）。根因多是行高 `<1` 或负字距套到了 CJK，见 `motion-kit.md` 第 2a 节。
5. **字体不靠本机**：实际样式/token 源中使用的非系统字体必须有获授权、跨平台的 Web 加载路径——作者本机常已装 `Arial Narrow / Bahnschrift / Cascadia` 等同名字体，问题不显形。确认浏览器实际字体来源符合设计（许可允许时可为站内 woff2，也可为获授权的字体服务），并验证网络失败/无本机字体时的回退；部署前核对自托管、子集化、CDN 使用与再分发许可。
6. 模拟 `prefers-reduced-motion: reduce`，确认核心内容仍可见、功能仍可用，且没有必要之外的持续动画。
7. 做一轮基础无障碍检查：语义地标与标题层级合理；控件/表单有可访问名称，校验错误能被感知；仅用键盘可完成主要任务；焦点样式清晰，弹层/菜单能正确管理焦点并用 `Escape` 关闭；文字与控件对比度、图片替代文本和触控目标合理；没有无效或冲突的 ARIA。项目已有 axe、Lighthouse 或同类检查时一并运行，但自动结果不能替代键盘与真实页面检查。

### 部署前检查点

1. 跑项目实际提供的构建、类型检查、测试或静态校验命令，并记录结果；无构建的静态站直接验证最终产物，不凭空添加 npm 工具链。
2. 检查 `<title>`、`meta description`、页面语言、绝对 canonical、Open Graph/Twitter 元数据和 favicon；`og:image` 应使用部署后可访问的绝对 URL，并核验实际尺寸与格式。
3. 检查生产配置、资源基路径和路由回退，不把本地地址、临时文件或仅开发环境可用的资源带入产物。
4. 若使用第三方资源、组件或字体，确认许可证、授权范围、署名和 NOTICE 等交付要求已满足；特别核对字体的自托管、子集化、CDN 使用和再分发权限。
5. 检查 Git 变更、`.env*`、静态产物和客户端 bundle，不让令牌、服务端密钥、私有 API 地址或凭据进入仓库/浏览器；浏览器端只放可公开配置。秘密值使用托管环境变量或服务端边界，扫描/报错时不要把实际凭据打印进日志。
6. 检查公开图片、缩略图与分享卡的 EXIF/GPS、设备、作者、软件和注释字段；按隐私清单移除敏感元数据，同时保留许可证或法律要求的版权信息。
7. **公开前确认页面上的私人联系方式**：页面若显示了简历里的手机号、微信号、身份证号、家庭住址等私人信息，在对外发布前先问用户是否要隐藏——不擅自替他公开，也不默认替他删（见铁律 7）。仅本地/私有交付时不必处理。

靠真实渲染判断，不靠回读源码。部署获授权并完成后，再对生产 URL 重跑基础检查点和部署前元数据检查。

## 性能清单

- 先确认产物复杂度与所选技术路径匹配；静态站不应只为通用淡入引入 React 或运行时动效库。
- React 路径：沿用项目已有动效方案；新增依赖时按所选版本的官方入口和按需加载方式实现，不把某个历史包名或 API 当作永久契约。
- 使用 Canvas/WebGL 时：DPR 上限（Canvas `≤2`、WebGL `≤1.75`）；密度按视口面积算、触屏减半；元素滚出视口用 `IntersectionObserver` 暂停，标签页/文档切后台用 `visibilitychange` 暂停 rAF。固定全屏背景不会因页面滚动离屏，但仍要处理后台暂停。
- 使用 three.js / paper-shaders 时：动态 import + 懒加载，不进主 bundle。
- **视频背景 Hero**（设计师作品集常见）：`muted playsinline autoplay`（否则移动端不自动播）+ `poster` 首帧图防白屏；控制体积（压缩、合适编码，必要时按连接/视口懒加载）；移动端或弱网降级为静帧/首帧图，别硬灌大视频；`prefers-reduced-motion` 下暂停并显示 poster。
- **滚动监听**：别裸挂高频 `scroll` 事件做布局计算；用 `IntersectionObserver`、rAF 节流或 ScrollTrigger 自带调度，避免每帧强制 reflow。
- 图片：别把用不到的塞进 `public`（每次构建都打包）；大图压缩。
- **首屏加载（LCP）**：盯首屏关键资源体积——视频/大图/字体是常见拖累；首屏主图/视频优先加载、非关键资源懒加载，并核对生产构建的首屏 JS 体积与关键渲染路径。
- 跑所选技术路径或现有项目实际提供的构建/校验命令，并检查产物体积；React/Vite 路径通常是 `npm run build`，纯静态无构建项目则验证本地服务或直接产物，不凭空添加 npm 工具链。

## 上线就绪（SEO / 社交分享 / favicon）

公开求职/作品站部署前按实际需要补齐；本地成品不因缺少公开 URL 元数据而失败：
- **SEO meta**：`<title>`、`meta description`、语言、canonical。
- **社交分享卡**：Open Graph（`og:title`/`og:description`/`og:image`）+ Twitter card；`og:image` 做 1200×630。
- **favicon 全套**：SVG + 各尺寸 PNG + apple-touch-icon。
- **JSON-LD `Person`（条件项）**：仅用于真实、公开、适合被索引且用户希望被搜索引擎理解为个人身份的页面。虚构人物、私密页面或不希望被索引的站点默认不输出可能误导的 `Person` 数据；必要时先确认更合适的结构化数据或完全省略。

## 部署（必须显式授权）

默认交付到“本地可运行”；用户要求后再做到“部署就绪”。上线就绪也不等于已经获得外部操作授权。创建仓库、提交、推送、改变仓库可见性、连接托管商、发布生产环境和修改 DNS 都是独立的外部状态变更，执行前确认用户明确要求及目标。

1. **保留现状**：已有 Git/VCS、远端仓库、私有仓库、分支策略和托管配置时沿用，不重新 `git init`、不换远端、不公开源码、不迁移托管。没有仓库时，先确认是否需要创建以及公开/私有属性。
2. **选择托管**：优先使用用户现有平台；没有既定平台时，再根据受众区域、项目运行时、预算、团队权限和实测结果选择。Cloudflare Pages、Vercel、Netlify、GitHub Pages 等都只是候选，不固定推荐某一家。
3. **区域与合规**：目标受众包含中国大陆时，以实际网络测试、备案/合规条件、域名和供应商可用性为依据，不对任何普通全球托管服务作笼统的大陆可达性承诺。
4. **认证与域名**：登录授权通常由用户本人在浏览器完成；不要索取或记录凭据。绑定域名、修改 DNS、切换生产流量前再次确认具体域名和变更内容。
5. **线上回归**：部署完成后对生产 URL 重跑真实渲染、元数据、资源路径和关键 CTA 检查；若失败，按现有发布策略回滚本轮变更，或在缺少既定策略时先向用户说明影响并确认恢复方案。

## 改版诊断（用户说"不好看/太单调"时）

**别把设计题甩回去问用户**——自己看页面，给具体诊断，按分级先小后大改。

先做一张 **反应 → 原因 → 修法** 表，自问：

| 用户反应 | 大概率原因 | 先试的修法 |
|---|---|---|
| "太单调 / 交互太少" | 内容主次、区块关系或操作反馈不清 | 先强化信息节奏和静态连贯性；确有访问者任务时，再按 `interview.md` Q3/Q6 加一条主动线或必要反馈，不默认加滚动动画 |
| "太商务 / 太装逼 / 像模板" | 撞了 AI 通病版式（铁律 9），缺少个人证据或设定支撑 | 回到参考和证据重做视觉语言；只有隐喻有至少两个独立信号支撑时才采用 |
| "还是有点像 / 换个人也能用" | 结构、任务或视觉语法同质：只换了职业外壳 | 过铁律 10 的换人 / 去皮 / 职业刻板三层自检；差异至少落到信息结构、Hero 构图、访问者任务/签名表达、贯穿动线/视觉语法中的两项 |
| "更丑了 / 麻麻赖赖" | 某个效果参数没调好、或堆叠过度 | 收敛：减数量、调动效预设/配色 token，别再加 |
| "不是我的风格" | 气质方向没对齐 | 回参考站校准张扬度，重定项目实际使用的设计 token 与样式源 |
| "这里不对 / 不居中" | 具体布局 bug | 通过真实渲染截图定位，单点修 |

**分级改版，先小后大**：润色（配色/间距/字重）→ 单个区块 → 视觉风格 → 动效体系 → 结构 → 整站翻转。能小改解决就别推倒；但真实项目里也发生过 3 次推倒重来——**当小改反复无效时，果断走结构级或翻转级**，并跑一次 grilling（`interview.md` 阶段 3）把大改收敛成一个连贯方案。
